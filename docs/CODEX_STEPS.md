# AI桌面助手 Codex 研发步骤

> 用途：为 Codex 提供 **小步、稳定、可测试** 的研发执行序列。  
> 基线来源：`docs/EXECUTION_MANUAL.md`。  
> 执行目标：让每次开发都只完成一个小闭环，始终保持项目可运行、可验证、可回滚。

---

## 0. 总执行原则

Codex 在本仓库内开发时，必须遵守以下原则：

1. **以 `docs/EXECUTION_MANUAL.md` 为唯一执行基线。**
2. **每次只完成一个 Step，不提前实现后续能力。**
3. **先改契约，再改后端，再改前端。**
4. **所有新增字段必须先进入 `contracts/`。**
5. **不允许把路由逻辑散落到 UI / LLM / Tools。**
6. **每一步都必须有最少必要测试。**
7. **每一步完成后，项目必须仍可运行。**
8. **高风险动作必须进入 `actions/` 层，不得直接写入工具层。**
9. **不允许静默失败，所有失败都必须可感知、可定位。**
10. **每一步结束后同步更新相关文档。**

---

## 1. 单步完成定义（Definition of Done）

每个 Step 完成时，必须同时满足：

- 功能已实现，且仅覆盖当前 Step 范围。
- 主流程可运行。
- 失败路径可观察、可提示。
- 相关契约已同步。
- 最少必要测试已补齐并通过。
- README / ARCHITECTURE / API_CONTRACT 有必要时已更新。
- 没有顺手引入未规划的大改动。

---

## 2. Codex 开发节奏

推荐节奏：

- **一次只发 1 个 Step 给 Codex**
- **一个 Step 只解决一个主问题**
- **一个 Step 最多新增一个主端点或一个主能力**
- **先非流式，后流式；先人控，后自动；先只读，后写入**

不推荐：

- 一次要求 Codex 完成整个 Phase
- 同时引入多个新依赖和多个新模块
- 在同一步里同时做契约重构、后端重构、前端重构、UI 重构

---

## 3. Step 总览

| Step | 名称 | 目标 | 风险等级 |
|---|---|---|---|
| 0 | 项目骨架初始化 | 建立最小可运行仓库结构 | 低 |
| 1 | 健康检查接口 | 建立首个真实 API 闭环 | 低 |
| 2 | 共享契约与类型生成 | 固定 DTO 真源 | 中 |
| 3 | Mock 聊天闭环 | 打通 UI → API → UI | 低 |
| 4 | 本地 LLM 非流式接入 | 完成本地 MVP 主链路 | 中 |
| 5 | Session 最小持久化 | 保存会话和消息 | 中 |
| 6 | 剪贴板工具接入 | 接入第一个工具能力 | 低 |
| 7 | 文件读取工具接入 | 接入第二个工具能力 | 中 |
| 8 | 路由状态展示 | 让 local / remote 透明可见 | 低 |
| 9 | 远程 LLM 接入 | 支持一键转在线 | 中 |
| 10 | SSE 流式聊天 | 增强体验，不改变主链路 | 中 |
| 11 | 真正的 Router 规则引擎 | 实现可解释、可覆盖路由 | 高 |
| 12 | 扩展能力阶段 | 截图 / 文档库 / actions | 高 |

## 4. 详细步骤

### Step 0：项目骨架初始化

#### 目标
建立最小可运行的桌面端与后端骨架，不接真实业务。

#### 本步产出
- `apps/desktop` 可启动
- `services/backend` 可启动
- 基础目录结构创建完成
- 基础脚本：
  - `scripts/dev_backend.sh`
  - `scripts/dev_desktop.sh`
- 空模块占位：
  - `modules/router/`
  - `modules/llm/`
  - `modules/tools/`
  - `modules/session/`
  - `modules/library/`
  - `modules/actions/`

#### 本步限制
- 不实现聊天
- 不实现模型调用
- 不实现工具读取
- 不实现自动路由

#### 本步测试
- 后端启动测试
- 前端启动测试
- 基础目录存在性检查（可选脚本）

#### 完成标准
- 前端可打开
- 后端可运行
- 无致命报错
- 仓库结构与执行手册一致

---

### Step 1：健康检查接口

#### 目标
建立第一个真实可调用 API，验证服务层基本结构正确。

#### 本步产出
- `GET /api/v1/health`
- 基础响应模型
- 统一响应格式
- 基础日志输出
- request_id / timestamp（可选但推荐）

#### 本步限制
- 不做 chat
- 不接 LLM provider
- 不接工具层

#### 本步测试
- API 集成测试：健康检查返回 200
- 响应字段断言测试

#### 完成标准
- curl 或前端请求可成功获取健康状态
- 响应字段符合契约
- 错误路径有明确处理

---

### Step 2：共享契约与类型生成

#### 目标
把后端 `contracts/` 固定为唯一真实来源，并让前端类型自动生成。

#### 本步产出
- `contracts/enums.py`
- `contracts/common.py`
- `contracts/chat.py`
- OpenAPI 导出脚本
- TS 类型生成脚本
- 前端消费生成后的类型

#### 本步限制
- 不改复杂业务逻辑
- 不做大规模 UI 重写
- 不手写复制 DTO

#### 本步测试
- schema 导出成功测试
- 类型生成成功测试
- 关键 DTO 字段一致性检查

#### 完成标准
- 新增字段时只需修改 contracts
- 前端不再手工维护重复 DTO
- 字段统一使用 `snake_case`

---

### Step 3：Mock 聊天闭环

#### 目标
先打通 UI → API → UI，不接真实模型。

#### 本步产出
- `POST /api/v1/chat`
- 前端输入框 / 发送按钮 / 消息列表
- 后端返回 mock 回复
- 最简消息结构
- `route_result` 暂时固定为 `local`

#### 本步限制
- 不接真实 LLM
- 不做流式
- 不做 session 落库

#### 本步测试
- chat API 集成测试
- UI 基础组件测试
- 输入后消息显示测试

#### 完成标准
- 用户发送一条消息后，能在界面看到 mock 回复
- 错误时有用户可见提示

---

### Step 4：本地 LLM 非流式接入

#### 目标
把本地 Ollama 接入聊天主链路，完成本地 MVP 主闭环。

#### 本步产出
- `modules/llm/base.py`
- `modules/llm/local_ollama.py`
- provider factory
- `/api/v1/chat` 接入真实本地调用
- 本地 provider 健康检查

#### 本步限制
- 只做非流式
- 只支持一个本地 provider
- 不做 remote provider
- 不做自动路由

#### 本步测试
- provider mock 测试
- 本地模型不可用测试
- chat API 集成测试

#### 完成标准
- 可以完成一轮真实本地问答
- 模型不可用时返回明确错误
- 日志能看出调用链路

---

### Step 5：Session 最小持久化

#### 目标
为聊天加入最小会话能力，支持历史消息保存。

#### 本步产出
- SQLite 基础存储
- `sessions` / `messages` 表
- session repository
- 基础 `create_session` / `append_message` / `list_messages`

#### 本步限制
- 不做复杂记忆
- 不做摘要压缩
- 不做跨会话检索

#### 本步测试
- 存储单元测试
- 新建会话测试
- 消息落库测试

#### 完成标准
- 同一 session 能读到历史消息
- 数据写入异常时有清晰错误

---

### Step 6：剪贴板工具接入

#### 目标
只接第一个工具能力，验证工具层调用链路。

#### 本步产出
- `GET /api/v1/tools/clipboard`
- 工具层 clipboard reader
- 前端“插入剪贴板”按钮
- `context_items` 支持 clipboard

#### 本步限制
- 不做文件读取
- 不做截图
- 不做写剪贴板

#### 本步测试
- clipboard 单元测试
- API 集成测试
- 前端上下文插入测试

#### 完成标准
- 用户可以将剪贴板文本插入上下文
- 空剪贴板时能明确提示

---

### Step 7：文件读取工具接入

#### 目标
接入文本文件读取能力，支持拖拽或导入。

#### 本步产出
- `POST /api/v1/tools/files/read`
- 文件读取器
- 路径验证与大小限制
- 前端文件拖拽 / 选择
- `context_items` 支持 file

#### 本步限制
- 只读
- 只处理文本类文件
- 不做 PDF 深解析
- 不做目录索引

#### 本步测试
- 文件存在 / 不存在测试
- 过大文件测试
- 二进制文件测试
- 前端导入行为测试

#### 完成标准
- 文本文件可进入上下文
- 越界或失败访问有明确错误

---

### Step 8：路由状态展示

#### 目标
把 local / remote / hybrid 的状态清楚展示在 UI 中。

#### 本步产出
- `StatusBadge` 或等价组件
- 消息级 `execution_mode` 展示
- 简单 `reason_text` 展示
- 前端状态同步逻辑

#### 本步限制
- 暂不做真正规则引擎
- route_result 可先由简单函数生成

#### 本步测试
- 状态组件测试
- 响应映射测试
- 聊天消息渲染测试

#### 完成标准
- 用户能清晰看到当前请求的执行模式
- 模式信息不是只存在日志里

---

### Step 9：远程 LLM 接入（一键转在线）

#### 目标
先支持用户手动切到在线模式，再考虑自动路由。

#### 本步产出
- `modules/llm/remote_openai.py`
- `requested_mode=remote`
- 前端“转在线”按钮
- 基础 remote provider 配置

#### 本步限制
- 不做自动路由
- 不做混合模式
- 不做搜索编排

#### 本步测试
- remote provider mock 测试
- 认证失败测试
- UI 模式切换测试

#### 完成标准
- 当前任务可明确切换为 remote 执行
- UI 能显示当前已走在线模型

---

### Step 10：SSE 流式聊天

#### 目标
增强交互体验，但不改变主功能边界。

#### 本步产出
- `POST /api/v1/chat/stream`
- 统一 SSE 事件：
  - `routing`
  - `chunk`
  - `done`
  - `error`
- 前端 streaming 渲染逻辑

#### 本步限制
- 不做复杂中断恢复
- 不做多路并发流合并

#### 本步测试
- SSE API 测试
- chunk 顺序测试
- 前端流式渲染测试

#### 完成标准
- 首 chunk 能及时展示
- 流中断时有清晰错误
- 非流式主链路不被破坏

---

### Step 11：真正的 Router 规则引擎

#### 目标
实现集中、可解释、可覆盖的任务路由逻辑。

#### 本步产出
- `RouteResult`
- `TaskType`
- 结构化规则 DSL 或纯 Python 规则函数
- 至少 5 条基础规则：
  - privacy_local
  - search_remote
  - long_context_hybrid
  - simple_rewrite_local
  - complex_code_remote
- 用户覆盖逻辑

#### 本步限制
- 禁止 `eval`
- 不把规则写入 UI / provider / tools
- 先做最小规则集，不做大而全分类器

#### 本步测试
- Router 单元测试（建议 15+ 场景）
- 用户覆盖测试
- 冲突规则优先级测试

#### 完成标准
- 路由结果有理由说明
- 用户可手动覆盖
- 其他层不含散落路由逻辑

---

### Step 12：扩展能力阶段

#### 目标
在 MVP 稳定后，再进入截图、文档库、受控修改能力。

#### 推荐顺序
1. 截图输入
2. 历史会话查看
3. 文档库导入
4. 文档库检索
5. `actions/plan`
6. `actions/apply`

#### 本步限制
- 所有修改类能力必须进入 `actions/`
- 不允许直接把写入能力加进 `tools/`

#### 本步测试
- 新能力单元测试
- 接口集成测试
- 高风险确认流程测试

#### 完成标准
- 扩展能力不破坏既有 MVP 主链路
- 写操作全部走 plan → confirm → apply

```text

硬约束：
1. 以 docs/EXECUTION_MANUAL.md 为唯一执行基线。
2. 先改 contracts，再改后端，再改前端。
3. 不允许把路由逻辑写到 UI / LLM / Tools。
4. 不允许静默失败。
5. 不允许做未记录的大范围重构。
6. 完成后必须给出：
   - 变更文件列表
   - 运行方式
   - 已补充的测试
   - 剩余风险
   - 更新的文档列表

本步目标：
[填写当前 Step 的目标]

本步产出：
[填写当前 Step 的产出]

本步测试：
[填写当前 Step 的测试]

完成标准：
[填写当前 Step 的验收]
```

## 6. 首轮最推荐执行顺序

如果现在开始让 Codex 落地，建议先只执行前 5 步：

1. Step 0：项目骨架初始化
2. Step 1：健康检查接口
3. Step 2：共享契约与类型生成
4. Step 3：Mock 聊天闭环
5. Step 4：本地 LLM 非流式接入

原因：
- 这 5 步就能形成一个稳定的本地 MVP 主链路
- 每一步都小、独立、可回滚
- 任何一步失败都容易定位问题
- 不会过早进入自动路由、流式、扩展能力等高复杂度区间

---

## 7. 每步结束时 Codex 必须汇报的内容

每完成一个 Step，Codex 必须输出：

1. 本步完成了什么
2. 修改了哪些文件
3. 如何运行和验证
4. 新增了哪些测试
5. 当前已知风险是什么
6. 下一步最合理的 Step 是什么

---

## 8. 禁止事项

Codex 在任何 Step 中，不得：

- 跳过 `contracts/` 直接改前后端字段
- 同时实现多个未来 Step
- 在 UI 中硬编码路由策略
- 在 Provider 中私自决定 local / remote
- 在 `tools/` 中加入写文件、删除文件、覆盖文件能力
- 引入大范围无说明重构
- 因“顺手”改动整个目录结构

---

## 9. 一句话执行口令

给 Codex 的最简口令可以是：

> 按 `docs/CODEX_STEPS.md` 执行当前 Step，只完成本步，保持项目可运行，并补齐最少必要测试。
